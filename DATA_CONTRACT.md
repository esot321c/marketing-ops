# Data Contract

Marketing-Ops separates reusable system files from user-owned marketing context.

## Tenant-first data layout

Inside `data/`, the tenant is the top-level directory: `data/<tenant>/{setup,work,content,analytics}/...`.
A tenant is any top-level directory under `data/` that contains a `setup/` subdirectory; that is
the whole discovery rule the app uses to list tenants. Everything else at the top level of `data/`
(legacy personal folders such as `brands/`, `guides/`, `research/`, `calendar/`, the `tenants/`
name/id registry, plus `data/shared/`) is ignored by tenant discovery and is never touched by the
migration script.

`data/shared/` holds global, non-tenant files: cross-tenant registries and config that used to live
inside a type directory, for example `content-library.json` and `analytics-review.json`, plus the
global `intake.md` when it is not tenant-specific.

If you have data in the old layout (`data/setup/<tenant>/...`, `data/work/<tenant>/<type>/...`,
`data/content/<tenant>/...`, `data/analytics/<tenant>/...`, `data/analytics/imports/<tenant>/...`),
run `scripts/migrate-tenant-first.mjs` once to move each tenant's files under `data/<tenant>/...`
and the global registry files under `data/shared/`. The script is idempotent (running it twice is a
no-op) and refuses to overwrite a destination that already has different content; pass `--dry-run`
to preview the moves first. It also aborts pre-flight, moving nothing, if a discovered tenant name
collides with a reserved directory (`shared`, `tenants`, `imports`, `brands`, `guides`, `research`,
`accounts`, `campaigns`, `assets`, `generated`) or is not a valid lowercase-kebab tenant id.

## System Layer

These files are reusable project logic and may be updated by upstream changes:

- `apps/dashboard/`
- `skills/`
- `templates/`
- `.codex-plugin/plugin.json`
- `README.md`
- `DATA_CONTRACT.md`
- `package.json`
- `scripts/migrate-tenant-first.mjs`

## Public Example Layer

These files are safe to commit and share:

- `data.example/tenants/*.json`
- `data.example/<tenant>/setup/*.json`
- `data.example/shared/`
- anonymous campaigns
- anonymous content examples
- template account inventories

## User Configuration Layer

These files personalize the system for a specific company, founder, or marketing motion:

- `data/brands/*.md`
- `data/tenants/*.json`
- `data/accounts/*.json`
- `data/campaigns/*.md`
- `data/competitors/*.md`
- `data/keywords/*.md`
- `data/voice/*.md`
- `data/<tenant>/setup/` (per-tenant Init state, intake, voice, and the tenant's **design system**: `design-system/tokens.json` + `design-system/previews/*.html`)
- `data/shared/` (global, non-tenant registries and config)
- `strategy/*.md` (optional, user-created local folder)
- `docs/*` (optional, user-created local folder for your own architecture notes and plans)

These private folders do not ship with the repo. You create them locally during init; they are gitignored and never committed. See the `marketing-setup` skill for how to create them.

Per-tenant design systems are user-owned data, not system code. The app ships **generic, reusable** components (`apps/dashboard/src/design-system/`); each tenant's brand tokens and `@dsCard` previews live under `data/<tenant>/setup/design-system/` and are served to the dashboard at runtime.

Agents may edit these files when the user asks them to update strategy, content, voice, or planning context.

## Generated Marketing Work

These files are generated by workflows and should be treated as user-owned output:

- `data/generated/`
- `data/<tenant>/work/<type>/*.md` (per-tenant generated marketing outputs: campaigns, strategy, keywords, research, analytics)
- `data/<tenant>/analytics/posts.json` (structured per-post analytics captures; fed by the XLSX import drop folder `data/<tenant>/analytics/imports/` and by manual captures the agent records)
- `output/`
- exported carousels
- exported scripts
- publishing queues
- analytics reports

## Local Tooling

These files should not be committed:

- `node_modules/`
- `.next/`
- `.env`
- `.env.local`
- local connector credentials

## Safety Rules

1. Do not publish externally without explicit user approval.
2. Do not send messages, outreach, DMs, or emails from Marketing-Ops.
3. Do not manage contacts, accounts, deal stages, or follow-up cadences here.
4. Hand off sales-ready context to your CRM instead of tracking it in Marketing-Ops.
5. Keep marketing source material and generated drafts reviewable as local files.
